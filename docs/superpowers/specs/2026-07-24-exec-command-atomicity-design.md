# exec_command 原子工作单元约束设计

## 背景

当前基线指令要求互不依赖的文件读取尽可能并行，但模型仍可能把多个读取或诊断任务通过 `;`、`&&` 等 shell 语法合并到一个 `exec_command.cmd`。这种调用虽然可以执行，但会让 Codex 客户端把整个调用展示为通用 `Run`，也会降低各项任务的输出隔离与可读性。

## 目标

约束所有 `exec_command` 调用，而不是只为 Skill 读取增加特例：

- 一个互不依赖的工作单元对应一个独立的 `exec_command` 工具调用。
- 同一轮存在多个互不依赖的工作单元时，必须拆分后并行发出。
- 禁止为了减少工具调用次数，使用 `;`、`&&`、`||`、循环、子 shell 或 `xargs` 将多个独立工作单元合并到同一个 `cmd`。
- 单个原子工作单元内部允许使用必要的管道、重定向和条件处理。

## 边界

允许的原子命令：

```bash
rg 'ERROR' gateway.log | tail -20
```

该命令的管道共同完成“读取最近的错误日志”这一项工作。

应拆分的独立命令：

```bash
sed -n '1,200p' first/SKILL.md
sed -n '1,200p' second/SKILL.md
```

两个文件读取互不依赖，应成为两个并行的 `exec_command` 调用，禁止通过 `;` 或 `&&` 合并。

## 实现

仅修改中英文基线指令的通用工具调用规则：

- 英文版和中文版表达相同的强制语义。
- 不在 Skill 章节增加重复规则。
- 不修改网关协议转换，不解析或拆分 shell 命令。
- 不改变 `exec_command`、`function_call`、`call_id` 或工具结果回灌行为。

## 验证

- 检查中英文规则语义一致。
- 确认规则同时覆盖文件读取、仓库诊断、测试等所有 `exec_command` 场景。
- 确认示例没有禁止原子任务内部合理使用管道。
- 运行仓库文档与基础门禁，确保没有引入格式或构建回归。

## 风险

基线指令只能约束模型行为，无法像运行时代码一样保证百分之百执行。更强的原子工作单元定义可以减少解释空间，但不应在网关层解析 shell 命令，否则会破坏工具调用的执行语义及结果关联。
